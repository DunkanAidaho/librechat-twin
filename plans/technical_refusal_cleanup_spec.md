# Техническая спецификация: Очистка истории сообщений от технических отказов

## 1. Цель
Автоматическое удаление сообщений ассистента, содержащих технические отказы (например, "Извините, новостей не найдено"), из истории сообщений, передаваемой модели. Это необходимо для предотвращения "зацикливания" модели на отказах, когда в текущем запросе уже имеется свежий RAG-контекст, позволяющий дать содержательный ответ.

## 2. Критерии фильтрации (Селективность)
Сообщение удаляется из истории только при одновременном выполнении следующих условий:
1. **Роль сообщения**: `assistant`.
2. **Паттерн отказа**: Текст сообщения соответствует списку технических отказов (метод `isTechnicalRefusal`).
3. **Наличие контекста**: В текущем запросе присутствует "свежий" RAG-контекст (`hasFreshContext`).

## 3. Список паттернов технических отказов
Метод `isTechnicalRefusal` должен распознавать следующие фразы (регистронезависимо):
- "не могу"
- "нет новост"
- "жду блок"
- "пришлите новости"
- "загрузка контекста завершена"
- "в текущем диалоге нет"
- "[...truncated...]"
- "извините, новостей не найдено"
- "информации не найдено"

## 4. Точки внедрения

### 4.1 [`api/server/services/agents/MessageHistoryManager.js`](api/server/services/agents/MessageHistoryManager.js)
- **Метод `isTechnicalRefusal(text)`**: Проверка текста на соответствие паттернам.
- **Метод `processMessageHistory`**: Основная логика фильтрации `orderedMessages` перед дальнейшей обработкой (сжатием, ингестией и т.д.).

### 4.2 [`api/server/controllers/agents/client.js`](api/server/controllers/agents/client.js)
- **Метод `buildMessages`**: Определение флага `hasFreshContext` и передача его в `historyManager.processMessageHistory`.
- **Критерий `hasFreshContext`**:
  - Наличие токенов в `req.ragContextTokens > 0`.
  - ИЛИ наличие сущностей в `req.ragMultiStep.ragContext.entities.length > 0`.

## 5. Алгоритм работы в `processMessageHistory`
1. Если `hasFreshContext === true`:
   - Создать новый массив `filteredMessages`.
   - Итерировать по `orderedMessages`.
   - Если сообщение от ассистента И является техническим отказом -> пропустить (удалить).
   - Иначе -> добавить в `filteredMessages`.
   - Логировать количество удаленных сообщений.
2. Если `hasFreshContext === false`:
   - Использовать исходный массив `orderedMessages`.

## 6. Безопасность реализации
- Использовать `filteredMessages` для всех последующих операций внутри метода.
- Не изменять исходную базу данных (удаление только из контекста текущего запроса).
- Обеспечить корректную обработку `null/undefined` в текстах сообщений.
